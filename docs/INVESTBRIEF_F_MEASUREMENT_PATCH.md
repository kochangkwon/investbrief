# INVESTBRIEF_F_MEASUREMENT_PATCH

> 패치 지시서 F-패치 — prompt_version 분리 집계 (지시서 F §7 미구현분)
> 대상: Claude Code / InvestBrief backend (Oracle Cloud ARM VM)
> 작성 기준일: 2026-07-02
> 우선순위: **P2** — StockAI 핫픽스(STOCKAI_V4_HOTFIX)와 무관하게 독립 진행 가능. 단, v2 프롬프트 알림이 쌓이기 시작한 지금 빨리 넣을수록 비교 표본이 온전해짐.
> 선행: 지시서 F 구현 병합 완료 (verify_prompts 빌더, MATERIALITY, 신선도 필터, prompt_version 기록 — 2026-07-02 검증에서 확인됨).

---

## 1. 개요 (Overview)

지시서 F 구현 검증 결과, `prompt_version`이 ThemeDetection에 **기록**은 되지만 30/60/90일 수익률 추적·집계(analytics)에 **분리 축이 없어** v1(구 프롬프트) 대비 v2(엄격 검증)의 효과를 측정할 수 없는 상태다. 기록만 있고 측정이 없으면 F의 존재 이유(타율 검증)가 성립하지 않는다. 본 패치는 기존 추적 인프라에 prompt_version group-by를 추가한다.

## 2. 배경 및 문제 정의

- `theme_radar_service`가 감지 저장 시 `prompt_version=PROMPT_VERSION("v2")` 기록 확인.
- `theme_alert_analytics`의 집계는 `theme_name` group-by만 존재 — prompt_version 축 부재.
- 18:05/18:15/18:25 크론(30/60/90일 수익률 추적)과 월간 리포트가 v1/v2 혼합 집계 → F 적용 전후 비교 불가.

## 3. 근본 원인

지시서 F §7이 "기록"과 "집계" 두 단계였는데 기록만 구현됨.

## 4. 목표 / 비목표

**목표**: (1) 30/60/90일 추적 집계와 월간 리포트에 prompt_version 분리 제공, (2) 표본 부족 시 판정 유보 규칙 내장.
**비목표**: 프롬프트 자체 변경, StockAI 연동 스키마 변경 (기존 internal API 응답 불변 유지).

## 5. 변경 사항 상세

### 5.1 `theme_alert_analytics` 집계 확장
- 기존 theme_name 집계에 더해 **prompt_version별 집계** 함수 추가: 알림 수, 30/60/90일 평균 수익률, 양(+)수익 비율. `prompt_version IS NULL`은 `"v1"`로 표기(F 이전 데이터).
- 월간/추적 리포트 출력에 블록 추가:

```
[프롬프트 버전 비교]
v1 (~06월): N건 | 30일 평균 +x.x% | 양수 비율 x%
v2 (07월~): N건 | 30일 평균 +x.x% | 양수 비율 x%
※ v2 표본 30건 미만 — 판정 유보
```

### 5.2 판정 규칙 내장
- v2 표본 < 30건 → "판정 유보" 자동 표기 (조기 과잉반응 방지).
- 비교 지표는 **양수 비율(타율)을 1순위**로 — F는 후보 수를 줄이는 방향이므로 건수 감소는 실패가 아님을 리포트 문구에 명시.

### 5.3 (정합) DoD 문구 정정
지시서 F DoD의 "radar/discovery 단일 빌더" 항목 관련: discovery의 테마 발굴 프롬프트는 용도가 달라(종목 검증이 아닌 테마 후보 발굴) 별도 유지가 타당함을 `docs/`에 1줄 기록 — 후속 세션이 불필요한 통합 작업을 벌이지 않도록.

## 6. DB 스키마
변경 없음 (prompt_version 컬럼 기존재).

## 7. 연동 영향
StockAI internal API 응답 불변. 리포트(텔레그램/월간)만 확장.

## 8. 텔레그램 / 웹 반영
- 텔레그램: 월간 리포트에 §5.1 블록.
- 웹(있는 경우): analytics 화면에 버전 필터 — 없으면 생략 가능(텔레그램이 1차 소비 채널).

## 9. 테스트 계획
1. NULL→"v1" 매핑 단위 테스트.
2. v2 표본 0건/29건/30건 → 유보/유보/판정 3케이스 렌더링.
3. 혼합 데이터 mock으로 버전별 평균·타율 계산 검증.

## 10. 배포 및 롤백

## 11. 운영 확인
